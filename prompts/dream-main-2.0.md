## Phase 1: 逐字稿回放（核心学习）

**目标**：像生物大脑回放经历一样，从今天的逐字稿中学习。

**操作步骤**：

1. **读逐字稿**：
   ```
   read ~/.openclaw/workspace-main/transcripts/main/{date}.md
   ```
   如果文件过大（>50K tokens），按需分段读取。也可以读最近 2-3 天的逐字稿来获取更多上下文。

2. **识别并记录**（写到临时文件 `memory/dream-learnings-{date}.md`）：

   - **正确决策**：今天哪个做法是对的？为什么？
   - **错误决策**：今天哪里做错了？怎么改进？
   - **新知识**：今天学到了什么新东西？（技术、工具、技巧）
   - **用户偏好**：用户今天表达了什么喜好/反感和？
   - **能力笔记**：今天完成某个任务用了什么好方法/框架？提炼为可复用的方法论

3. **能力笔记写入**：
   - 提炼出的方法论写入 `memory/ideas/{date}-capability.md`
   - 格式：标题（任务类型）+ 框架步骤 + 适用场景 + 注意事项
   - 这是"怎么做更好"的知识，区别于规则（"不该做什么"）

4. **重点标记**：
   - 如果某个教训非常重要（可能导致严重后果），标记为 🔴 红线
   - 如果是工作流层面的经验，标记为 🟡 规则
   - 如果是技巧类，标记为 🟢 经验

**输出格式**：
```markdown
# 学习笔记 {date}

## 🔴 红线类
- （必须遵守，否则严重后果）

## 🟡 工作流类
- （应该遵守，提高质量）

## 🟢 经验技巧类
- （提高效率，不遵守也没事）

## 用户偏好
- （记住用户的偏好）
```

## Phase 2: 知识提炼 + Facts 写入

**目标**：把学习笔记整合到长期记忆，同时写入 facts 文件（ADD-only 模式）。

**操作步骤**：

### 2.1 Facts 提取与写入（ADD-only）

借鉴 Mem0 的 ADD-only 模式：每条新记忆写成独立文件，不修改旧文件。

1. **读实体列表**：`read memory/entities.json`

2. **从 Phase 1 学习笔记中提取事实**（每条事实一个文件）：
   - 文件路径：`memory/facts/{date}T{HHMM}-{简短摘要}.md`
   - 格式：
   ```markdown
   ---
   date: {YYYY-MM-DD}
   entities: [{从 entities.json 匹配的实体列表}]
   tags: [{从 entities.json tags 匹配的标签}]
   category: {红线|规则|经验|偏好|知识}
   source: "{对话主题}"
   status: active
   ---
   
   {事实内容，1-3句话}
   ```

3. **实体匹配规则**（不调 LLM，用文本匹配）：
   - 遍历 entities.json 中的所有实体
   - 如果事实内容中出现该实体名称 → 加入 entities 列表
   - 遍历 tags 中的关键词 → 命中则加入 tags

4. **ADD-only 原则**：
   - 只创建新文件，不修改已有 facts 文件
   - 如果发现与旧 fact 矛盾 → 在新 fact 的 category 标为"更新"，并在内容中引用旧 fact
   - 过期内容 → 在新 fact 中标记 `related: [{旧fact文件名}]`

### 2.2 MEMORY.md 索引更新

5. **读当前 MEMORY.md**：`read MEMORY.md`

6. **重要：MEMORY.md 是纯索引文件，不写具体内容。**
   每条索引格式：
   ```
   ### 主题标题 → `facts/文件名.md`
   一句话摘要
   ```

7. **索引更新规则**：
   - 新 facts 对应的主题在索引中不存在 → 在「索引」段落末尾追加新索引条目
   - 新 facts 对应的主题已存在 → 不修改索引（索引已指向同主题的 fact 文件）
   - 索引条目只包含：标题 + 文件路径 + 一句话摘要
   - **绝对不往 MEMORY.md 写具体知识内容、经验细节、规则正文**

8. **索引条目示例**：
   ```
   ### 新主题 → `facts/2026-04-25T0230-新主题.md`
   一句话描述这个主题的核心内容
   ```

9. **红线/规则的处理**：
   - 🔴 红线 → 只在 AGENTS.md「学来的规则」段落处理（Phase 3），不在 MEMORY.md 写
   - 🟡 规则 → Phase 3 处理
   - 🟢 经验 → 写入 facts/，MEMORY.md 只更新索引

## Phase 3: 规则进化

**目标**：从工作流规则中提炼可执行的行为规则，写入 AGENTS.md「学来的规则」段落。

**操作步骤**：

1. **读 AGENTS.md**：
   ```
   read ~/.openclaw/workspace-main/AGENTS.md
   ```

2. **从 Phase 2 的 🟡 规则中筛选候选**：
   - 只保留"可执行"的规则（比如"配置改完必须重启"）
   - 排除"知道什么"（这类信息已经在 MEMORY.md）

3. **写入格式**：
   ```markdown
   - [验证第N天] 规则描述 _(来源: YYYY-MM-DD 对话主题)_
   ```

4. **验证已有规则**：
   - 检查今天的工作中遵守了哪些规则
   - 如果违反 → 标记为 `[⚠️ 冲突]`，等待人工裁决
   - 如果连续 3 天验证通过 → 标记为 `[固化]`

5. **只写入「学来的规则」段落**：
   - 不要修改 AGENTS.md 的其他内容
   - 遵循写入指南（段落顶部有说明）

## Phase 3.5: 规则回检（验证闭环）

**目标**：验证固化规则是否真正被执行，形成学习闭环。

**操作步骤**：

1. **读取固化规则**：从 AGENTS.md「学来的规则」段落中提取所有 `[固化]` 规则

2. **回检方法**：
   - 搜索今天和近 7 天的逐字稿（`transcripts/main/{date}.md` 和 `memory/` 下的近期日志）
   - 逐条检查每条固化规则是否有被遵循的证据

3. **回检结果标记**：
   - ✅ **已执行**：有明确证据表明规则被遵循
   - ⚠️ **未触发**：今天没有相关场景，无法判断
   - 🔴 **违反**：有证据表明规则被违反

4. **输出回检报告**（嵌入 Phase 5 汇报中）：
   ```markdown
   ### 规则回检
   | 规则 | 状态 | 证据 |
   |------|------|------|
   | 配置变更后同步 MEMORY.md | ✅ | 今天改了 X 配置，已同步 |
   | 每步先论证再执行 | ⚠️ | 今天无复杂操作 |
   ```

5. **休眠机制**：
   - 如果某条规则连续 3 次 Dream 回检都是"未触发"（即 3 周没相关场景）→ 标记为 `[💤 休眠]`
   - 休眠规则仍保留，但不计入活跃规则数
   - 如果后续场景再次出现，自动恢复为 `[固化]`

6. **违反处理**：
   - 发现违反 → 在 Phase 5 汇报中明确指出
   - 不自动降级，等待用户裁决
   - 如果用户确认规则已不适用 → 标记为 `[❌ 废弃]`

7. **持久化回检结果**：
   - 写入 `memory/dream-review-{date}.md`
   - 格式：
     ```markdown
     # 规则回检 {date}
     | 规则 | 状态 | 连续未触发次数 | 备注 |
     |------|------|---------------|------|
     ```
   - 读取最近 7 天的 `dream-review-*.md` 来判断连续状态
   - 这是下次回检的历史依据，不写这里就无法判断"连续3次"

**操作步骤**：

1. **读采集数据**：
   ```
   read ~/.openclaw/workspace-main/memory/dream-collect-{date}.md
   ```

2. **分级处理**：

   **🟢 可自动修（可逆+低风险）**：
   - 清理临时文件
   - 归档 >7 天的日志到 `memory/archive/`
   - 修正明显的配置错误（如过期的 user_id）
   - 执行命令用 `exec`
   - **标记过期 facts**：扫描 facts/ 中 status=active 的文件，如果内容已明显过时（如"MiMo 已移除"但 MiMo 早已不存在），将 status 改为 expired，不删除文件
   - **检测矛盾 facts**：如果两个 facts 内容矛盾（如"减肥中" vs "放弃减肥"），在新 fact 中标注 supersede: {旧fact文件名}

   **🟡 需确认后修（中风险）**：
   - 修脚本 bug
   - 改 cron 配置
   - 修正 MEMORY.md 中的描述错误
   - **不直接修改，写修复方案到待办列表**

   **🔴 只报不碰（高风险）**：
   - 改模型配置
   - 删除文件
   - 改 `openclaw.json` 核心配置
   - **只记录到 MEMORY.md「待办」**

## 安全规则

1. **永远不要**：
   - 删除 `memory/` 下的任何 `.md` 文件
   - 修改 `AGENTS.md` 的「学来的规则」段落之外的内容
   - 修改 `SOUL.md`
   - 删除 `sessions/` 下的文件

2. **可以修改**：
   - `MEMORY.md`：只追加/更新，不删除已有内容
   - `AGENTS.md`：只追加到「学来的规则」段落
   - 脚本文件：只修复 bug，不改变功能逻辑

3. **遇到不确定的事**：
   - 标记到 MEMORY.md「待办」
   - 在汇报中说明"需要人工确认"
