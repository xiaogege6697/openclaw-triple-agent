# Dream 2.0 系统设计文档

## 设计理念

Dream 是 Agent 的**自动学习与进化系统**。每天凌晨自动运行，从当天的工作经历中提炼经验、固化规则、维护系统。

核心类比：Agent 的"做梦"≈ 人类的睡眠记忆巩固。白天积累的短期记忆，在"做梦"时被筛选、提炼、整合为长期记忆和行为规则。

## 灵感来源

| 生物机制 | 系统对应 | Dream 阶段 |
|----------|---------|------------|
| **海马体回放** | 逐字稿回放 | Phase 1 |
| **记忆巩固** | 短期→长期（facts 写入） | Phase 2 |
| **突触可塑性** | 规则进化（行为矫正） | Phase 3 |
| **类淋巴清除** | 系统自愈（归档/清理） | Phase 4 |
| **意识恢复** | 汇报推送 | Phase 5 |

## 五阶段流水线

### Phase 1: 逐字稿回放（核心学习）

**输入**：`transcripts/main/{date}.md`（agent 每轮对话后实时写入）

**处理**：
1. 读取当日逐字稿（过大时分段读取）
2. 识别正确决策、错误决策、新知识、用户偏好
3. 按严重程度分类标记：
   - 🔴 红线（违反会导致严重后果）
   - 🟡 规则（提高质量的约束）
   - 🟢 经验（提高效率的技巧）
4. 提炼可复用方法论 → 写入 `memory/ideas/`

**输出**：`memory/dream-learnings-{date}.md`

### Phase 2: 知识提炼 + Facts 写入

**核心机制：ADD-only 事实库**（借鉴 Mem0）

- 每条新知识写为独立文件：`memory/facts/{date}T{HHMM}-{topic}.md`
- 只建新文件，不修改旧文件
- 实体匹配：遍历 `entities.json`，文本匹配关联实体
- 矛盾处理：新 fact 标注引用旧 fact，不删除旧 fact
- 过期标记：`status: expired`，不删除

**MEMORY.md 更新**：
- MEMORY.md 是纯索引文件（不写具体内容）
- 新 facts 对应主题 → 追加索引条目
- 格式：`### 主题标题 → facts/文件名.md` + 一句话摘要

### Phase 3: 规则进化

**从 🟡 规则中提炼可执行的行为规则**，写入 AGENTS.md「学来的规则」段落。

**规则生命周期**：

```
[验证第1天] 新规则
    ↓ 连续3天出现在 Dream 报告中，且无冲突
[固化] 固化规则
    ↓ 后续对话中被违反
[⚠️ 冲突] 冲突规则 → 等待人工裁决
    ↓ 连续3次回检"未触发"
[💤 休眠] 休眠规则 → 场景出现时自动恢复
```

### Phase 3.5: 规则回检（验证闭环）

**验证固化规则是否真正被执行**：

1. 读取所有 `[固化]` 规则
2. 搜索近 7 天逐字稿和日志
3. 逐条标记：✅ 已执行 / ⚠️ 未触发 / 🔴 违反
4. 休眠机制：连续 3 次"未触发" → `[💤 休眠]`
5. 违反处理：报告但不自动降级，等人工裁决

**输出**：`memory/dream-review-{date}.md`（持久化回检历史，用于判断连续状态）

### Phase 4: 系统巡检 + 自愈

**分级处理机制**：

| 级别 | 示例 | 操作 |
|------|------|------|
| 🟢 自动修 | 归档旧日志、清理临时文件、标记过期 facts | 直接执行 |
| 🟡 待确认 | 修脚本 bug、改 cron 配置 | 写方案，等用户确认 |
| 🔴 只报告 | 改模型、删文件、改核心配置 | 只记录到待办 |

### Phase 5: 汇报推送

向用户推送今日 Dream 报告，包含：
- 系统状态（Gateway/磁盘/Cron）
- 学习笔记（各类别数量）
- 记忆更新（新增/过期/矛盾）
- 规则进化（新增/固化/冲突）
- 规则回检（已执行/未触发/违反/休眠）
- 自愈操作（自动修复/待确认/只报告）

**推送方式**：使用 message 工具主动发送微信消息，确保用户收到独立播报。

## 学习笔记分类

```markdown
# 学习笔记 {date}

## 🔴 红线类
- （必须遵守，否则严重后果）

## 🟡 工作流类
- （应该遵守，提高质量）

## 🟢 经验技巧类
- （提高效率，不遵守也没事）

## 用户偏好
- （记住用户的偏好）

## 💡 能力笔记
- （可复用的方法论框架）
```

## Facts 文件格式

```markdown
---
date: {YYYY-MM-DD}
entities: [{从 entities.json 匹配的实体列表}]
tags: [{匹配的标签}]
category: {红线|规则|经验|偏好|知识}
source: "{对话主题}"
status: active
---

{事实内容，1-3句话}
```

## 四级查询体系

Agent 遇到不确定信息时，按成本递进查询：

| 级别 | 方法 | 成本 | 适用场景 |
|------|------|------|----------|
| L1 | 内部记忆对比 | 零成本 | 默认先跑 |
| L2 | web_fetch 1-2 次 | 低 | 拿不准时 |
| L3 | 多源交叉验证 | 中等 | 信息矛盾时 |
| L4 | 完整搜索+读源码+跑测试 | 高 | 结论错误不可逆时 |

判断标准：**结论错了修复成本多大**。

## 安全规则

1. **永远不要**：
   - 删除 memory/ 下的任何 .md 文件
   - 修改 AGENTS.md「学来的规则」段落之外的内容
   - 修改 SOUL.md
   - 删除 sessions/ 下的文件

2. **可以修改**：
   - MEMORY.md：只追加/更新索引
   - AGENTS.md：只追加到「学来的规则」段落
   - 脚本文件：只修复 bug

3. **不确定的事**：
   - 标记到 MEMORY.md「待办」
   - 在汇报中说明"需要人工确认"
